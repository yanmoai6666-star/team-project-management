# 标签颜色标准化文档

## 背景介绍
随着团队项目管理系统（team-project-management）的不断发展，项目中使用的GitHub标签逐渐增多且颜色使用混乱，导致团队成员在识别和管理问题时效率降低。为了建立统一、规范的标签颜色体系，提高项目管理的可视化效果和协作效率，特制定本标签颜色标准化规范。

## 标签分类体系
本项目的标签按照以下五个维度进行分类：

1. **问题类型**：用于标识Issue或PR的类型，如bug、功能请求等
2. **功能类型**：用于标识功能相关的标签
3. **优先级**：用于标识任务的重要性和紧急程度
4. **状态**：用于标识任务或问题的当前状态
5. **组件**：用于标识问题所属的代码模块或系统组件
6. **其他类型**：不适合上述分类的特殊标签

## 颜色标准规则
标签颜色遵循以下标准化规则：

1. **问题类型标签**：使用红色系（#d73a4a、#fbca04等），便于快速识别问题
2. **功能类型标签**：使用蓝色系（#a2eeef、#1d76db等），表示功能相关
3. **优先级标签**：使用红/橙/黄三色系（#ff0000、#ff9900、#ffff00），直观反映紧急程度
4. **状态标签**：使用绿色系（#bfe5bf、#c2e0c6等），表示任务进展状态
5. **组件标签**：使用彩色系（#fca326、#80d8ff等），区分不同技术组件
6. **其他标签**：根据具体用途分配颜色

## 使用指南

### 1. 标签选择原则
- **问题类型标签**：每个Issue必须至少有一个问题类型标签
- **优先级标签**：所有任务类Issue必须标注优先级
- **状态标签**：根据任务进展及时更新状态标签
- **组件标签**：适用于需要明确技术归属的问题

### 2. 标签组合使用
- 一个Issue可以同时拥有多个不同分类的标签
- 例如：一个前端严重bug可以同时标注 `bug-critical`、`priority-high` 和 `component-frontend`

### 3. 标签管理流程
1. **创建Issue时**：根据问题性质选择合适的标签组合
2. **处理过程中**：及时更新状态标签（如从 `status-in-progress` 改为 `status-review`）
3. **解决问题后**：更新为 `status-done`，并根据情况添加其他标签

### 4. 标签维护责任
- **项目维护者**：负责创建、更新和删除标签
- **所有贡献者**：按照本规范正确使用标签
- **新人贡献者**：可关注 `good-first-issue` 标签寻找适合入门的任务

### 5. 标签颜色修改
如需修改标签颜色，需：
1. 提交Issue讨论颜色变更的必要性
2. 获得项目维护者批准
3. 更新本文档中的颜色定义
4. 在GitHub中批量更新现有标签颜色

## 附录

### 颜色选择依据
所有颜色选择均参考GitHub官方标签颜色体系，确保：
- 足够的对比度，便于识别
- 颜色语义符合行业惯例
- 色盲友好，使用不同饱和度和明度区分

### 标签数量扩展
如需添加新标签，需：
1. 确定标签所属分类
2. 选择与分类相符的颜色系
3. 更新本文档
4. 在GitHub中创建新标签
